Research: Libraries and Frameworks for Building Futureboard Insert Plugins
Date: 2026-05-16

Scope:
Find practical libraries/frameworks that can help Futureboard Studio build built-in insert plugins faster than writing every DSP algorithm by hand in TypeScript.

Current project context:
- Web app: React + Vite + WebAudio.
- Current insert UI/state source of truth: `project.tracks[].inserts`.
- Current audio graph issue: inserts exist in state/UI, but playback audio is not routed through plugin DSP yet.
- Existing dependency: `@soundtouchjs/audio-worklet`.
- Near-term target: WebAudio insert DSP runtime.
- Long-term target: Rust/WASM/AudioWorklet/native service and SphereEngine DAUx-style audio engine.

Important selection criteria:
1. Can run in browser/Electron renderer.
2. Can be controlled from our own plugin UI.
3. Can be inserted into a WebAudio graph.
4. Realtime-safe enough for playback.
5. Does not force us into a totally different app architecture.
6. Can be bundled by Vite/Bun with reasonable effort.
7. Helps with EQ/delay/reverb/compressor/pitch/time/analyzer work.
8. License/commercial constraints are acceptable or clearly known before adoption.

============================================================
Executive Recommendation
============================================================

Best near-term stack:

1. Use native WebAudio nodes for the first working insert runtime.
   - BiquadFilterNode for Equz8.
   - DelayNode/GainNode/BiquadFilterNode/WaveShaperNode for UltraDelay.
   - ConvolverNode first pass for UltraVerb.
   - DynamicsCompressorNode first pass for FB-2A Comp.

2. Add a plugin DSP runtime abstraction now.
   - `track.inserts` -> `InsertDspRuntime` -> WebAudio node chain.
   - Keep React plugin UI separate from DSP factories.

3. Use Faust for custom DSP that native WebAudio cannot cover well.
   - Best candidate for higher-quality reverb, compressor, saturation, filters, modulation, FFT-ish DSP.
   - Faust can compile DSP to WebAssembly/WebAudio wrappers.

4. Study Web Audio Modules (WAM), but do not make it the only built-in plugin path yet.
   - WAM is useful if we want a plugin ecosystem/format.
   - Built-in plugins should still work through our own runtime.
   - Later: add WAM host support as an optional plugin format.

5. Keep SoundTouch for realtime pitch/time preview.
   - Already integrated.
   - Useful for clip pitch/time, not enough for EQ/reverb/delay/comp.

6. Consider Superpowered only if commercial licensing and binary SDK workflow are acceptable.
   - Strong realtime audio SDK, likely highest quality shortcut.
   - But it adds licensing/build/process constraints.

Fastest path for current bug:
Do not start by integrating a large third-party plugin framework.
First implement the internal WebAudio insert chain and native-node DSP. After Add Insert actually changes sound, use Faust/Superpowered/WAM selectively.

============================================================
Shortlist Table
============================================================

1. Native WebAudio Nodes
Use for: EQ, delay, simple reverb, compressor, filters, gain, pan, analyser.
Fit: Excellent for immediate built-in insert DSP.
Pros:
- Already available.
- No new dependency.
- Stable browser/Electron support.
- Easy to automate parameters.
- Works directly in current `Mixer`.
Cons:
- Some algorithms are basic.
- DynamicsCompressorNode is not an LA-2A-style opto comp.
- ConvolverNode IR reverb can be CPU/memory heavy if rebuilt too often.
Recommendation:
- Use immediately.

2. Faust WebAssembly/WebAudio
Use for: custom production DSP, reverb, compressor, saturation, filters, modulation.
Fit: Excellent medium-term.
Pros:
- Designed for DSP.
- Can compile to WebAssembly and WebAudio-compatible code.
- Large DSP ecosystem/examples.
- Keeps DSP code clearer than handwritten TS for complex effects.
Cons:
- Adds a DSP language/toolchain.
- Need build integration and parameter mapping.
- Generated wrappers need careful bundling/testing.
Recommendation:
- Adopt for custom DSP after the WebAudio insert runtime works.
- Good candidate for UltraVerb v2 and FB-2A v2.

3. Web Audio Modules (WAM)
Use for: plugin format/runtime, hosting external WebAudio plugins, future ecosystem.
Fit: Good for plugin architecture, not a quick DSP shortcut by itself.
Pros:
- Designed around WebAudio plugin modules.
- Helps define plugin lifecycle, processors, GUIs, automation.
- Useful if Futureboard wants third-party web plugins later.
Cons:
- Requires host/runtime integration.
- Third-party plugin availability may be limited compared with native VST.
- Does not automatically solve built-in plugin DSP quality.
Recommendation:
- Study and maybe support as optional host format.
- Do not block built-in plugin runtime on WAM.

4. Superpowered Web Audio SDK
Use for: commercial-quality realtime DSP, effects, time-stretch/pitch, players, analyzers.
Fit: Potentially excellent if license is acceptable.
Pros:
- Built for low-latency professional audio.
- WebAssembly-focused.
- Offers many ready audio primitives/effects.
- Likely strong performance.
Cons:
- Commercial/licensing constraints need review.
- Larger SDK integration.
- May not align with open internal plugin architecture.
Recommendation:
- Evaluate in a separate spike.
- Strong option if we want to buy time/quality instead of writing DSP.

5. Tone.js
Use for: quick WebAudio effects and musical utilities.
Fit: Good for prototyping, mixed for DAW production inserts.
Pros:
- Popular WebAudio framework.
- Has effects such as Reverb, FeedbackDelay, Compressor, EQ/filter utilities.
- Easy to experiment with.
Cons:
- Higher-level framework may impose its own timing/audio graph patterns.
- Could conflict with existing transport/mixer design if overused.
- Bundle size and abstraction overhead need checking.
Recommendation:
- Use as reference/prototyping source or for isolated devices only.
- Avoid making Tone.js the central engine.

6. Tuna.js
Use for: classic WebAudio effect nodes, quick prototypes.
Fit: Low-to-medium.
Pros:
- Purpose-built collection of WebAudio effects.
- Includes common effects like delay, reverb, compressor-ish effects, chorus, overdrive.
Cons:
- Older ecosystem.
- Maintenance/TypeScript/bundler compatibility may be weak.
- Quality may be prototype-level.
Recommendation:
- Use as reference only unless a quick test proves it works cleanly.

7. RNBO Web Export
Use for: exporting Max/RNBO patches to web effects.
Fit: Powerful but workflow-heavy.
Pros:
- Can build complex audio devices visually in RNBO.
- Web export can run in browser contexts.
- Great for rapid DSP ideation if Max/RNBO is part of the workflow.
Cons:
- Requires RNBO/Max workflow and licensing.
- Generated devices may not fit cleanly into our source-controlled plugin packages.
- Less ideal for core built-in plugins unless the team commits to RNBO.

Use for: modern audio DSP language and browser demos/worklets.
Fit: Interesting future option.
Pros:
- Designed for audio DSP.
- Has web-oriented execution examples.
- Could be a clean alternative to Faust for custom DSP.
Cons:
- Ecosystem and build integration need validation.
- May be less mature/widespread than Faust for web plugin work.
Recommendation:
- Keep on watchlist; spike only after Faust/native-node path is stable.

Fit: Interesting but likely architectural shift.
Pros:

- Can run audio in worklet-style environments.

Cons:

Native WebAudio Nodes
---------------------

Relevant APIs:
- `BiquadFilterNode`
- `DelayNode`
- `GainNode`
- `StereoPannerNode`
- `ChannelSplitterNode`
- `ChannelMergerNode`
- `ConvolverNode`
- `DynamicsCompressorNode`
- `AnalyserNode`
- `AudioWorkletNode`

Why this matters now:
The immediate problem is not lack of DSP algorithms. It is missing graph routing. Native nodes can prove the insert runtime quickly.

Recommended mapping:

Equz8:
- chain of `BiquadFilterNode`
- HP/LP/shelf/bell types map directly to native biquads

UltraDelay:
- `DelayNode` per channel
- feedback via `GainNode`
- tone shaping via `BiquadFilterNode`
- optional saturation via `WaveShaperNode`

UltraVerb:
- first pass `ConvolverNode` with generated IR
- `DelayNode` for pre-delay
- filters for low/high cut
- dry/wet gains

FB-2A Comp:
- first pass `DynamicsCompressorNode`
- dry/wet gains for parallel mix
- makeup/output trim with `GainNode`
- later replace with custom Worklet/Faust for opto behavior

Key implementation warning:
Do not rebuild whole graph on every slider change. Create nodes once and update AudioParams.

Official docs:
- Web Audio API: https://developer.mozilla.org/en-US/docs/Web/API/Web_Audio_API
- DelayNode: https://developer.mozilla.org/en-US/docs/Web/API/DelayNode
- AudioWorkletProcessor process: https://developer.mozilla.org/en-US/docs/Web/API/AudioWorkletProcessor/process

Faust
============================================================

What it is:
Faust is a functional DSP language that can generate audio plugins, C++, WebAssembly, WebAudio-compatible code, and other targets.

Why it fits Futureboard:
- We can write DSP in a dedicated language instead of complicated TypeScript.
- Generated WebAssembly can run in browser/Electron.
- Later the same DSP concepts can inform Rust/native implementations.

Good use cases:
- Reverb network
- Opto compressor detector/gain computer
- Saturation/tape/tube curves
- Modulation effects
- FFT/spectral experiments
- Filters beyond simple biquads

Integration idea:
- Keep plugin package structure:
  - `plugins/UltraVerb/Core`
  - `plugins/UltraVerb/Editor`
  - `plugins/UltraVerb/Dsp/faust`
- Add build step that compiles `.dsp` to a WebAudio/WASM module.
- Wrap generated processor in `InsertAudioNode` interface.

Risks:
- Build pipeline complexity.
- Need generated artifacts policy.
- Need parameter metadata mapping.
- Need performance validation in Vite/Electron.

Recommendation:
Use after Phase 1 native WebAudio runtime works. Faust is likely the best "write less custom TS DSP" option for production-quality custom effects.

Sources:
- Faust main site: https://faust.grame.fr/
- Faust WebAudio/WebAssembly documentation/search entry: https://faustdoc.grame.fr/

Web Audio Modules (WAM)
============================================================

What it is:
WAM is a plugin-module architecture for WebAudio. It aims to make web audio plugins loadable and hostable with audio processors and GUIs.

Why it fits Futureboard:
- Futureboard is building a DAW-like app.
- A plugin format/runtime could let us load third-party web plugins later.
- WAM concepts align with insert devices, automation, processor lifecycle, and plugin GUIs.

Where it helps:
- External plugin hosting.
- Standardized plugin metadata/lifecycle.
- Separating plugin processor from UI.

Where it does not help immediately:
- It will not automatically create a good Equz8/UltraVerb/UltraDelay/FB-2A.
- It may slow down the immediate fix if adopted too early.

Recommended architecture:
- Built-in plugins use Futureboard `InsertDspRuntime`.
- Later add `WamInsertNode` as one insert factory type.
- WAM plugin can be loaded as one device in `track.inserts`.

Sources:
- Web Audio Modules SDK GitHub: https://github.com/webaudiomodules/sdk
- Web Audio Modules organization: https://github.com/webaudiomodules

Superpowered Web Audio SDK
============================================================

What it is:
Superpowered is a high-performance audio SDK with JavaScript/WebAssembly support for web audio applications.

Why it fits:
- Strong candidate for serious realtime DSP without writing everything ourselves.
- Can cover professional audio primitives, effects, analysis, playback, and time/pitch features.

Good use cases:
- High-quality filters/effects.
- Pitch/time processing.
- DJ/DAW realtime operations.
- Analyzer/metering.

Risks:
- Licensing/cost.
- SDK integration and binary artifacts.
- Need to check if each required effect exists and how to wrap it in our insert runtime.

Recommendation:
Run a separate paid/commercial viability spike before adopting. If licensing is acceptable, this could save significant DSP development time.

Source:
- Superpowered JS Audio Library & SDK: https://superpowered.com/js-audio-library-sdk

Recommended Implementation Plan
============================================================

Phase A: Fix the real blocker first
1. Implement `InsertDspRuntime`.
2. Refactor `Mixer` track graph to include insert chain points.
3. Sync `project.tracks[].inserts` into the runtime.
4. Unknown plugin types pass through.
5. Empty chain sounds identical to current behavior.

Phase B: Native WebAudio built-ins
1. Equz8 with `BiquadFilterNode`.
2. UltraDelay with `DelayNode`, feedback `GainNode`, feedback filters.
3. UltraVerb with `ConvolverNode` generated IR.
4. FB-2A with `DynamicsCompressorNode` first pass.

Phase C: Evaluate libraries after audio is actually connected
1. Faust spike:
   - build one small DSP, for example saturation or Schroeder reverb.
   - wrap it as `InsertAudioNode`.
2. Superpowered spike:
   - test licensing.
   - test one filter/effect in Vite/Electron.
3. WAM spike:
   - host one example module as an insert.

Phase D: Upgrade quality
1. Replace simple Convolver reverb with Faust/Worklet reverb if needed.
2. Replace native compressor with Faust/Worklet opto compressor if needed.
3. Add analyzer/meter paths with transient state, not project state.

============================================================
Final Recommendation
============================================================

The best way to reduce development time without creating a bigger architecture problem:

1. Build the insert runtime with native WebAudio first.
2. Use native nodes for the first production-usable versions.
3. Add Faust as the main "write less custom TypeScript DSP" path.
4. Keep WAM as a future plugin-hosting layer, not the foundation of built-in plugins.
5. Evaluate Superpowered separately if commercial SDK use is acceptable.

This keeps Futureboard moving quickly:
- Add Insert starts changing sound immediately.
- UI remains ours.
- `track.inserts` remains the truth.
- DSP can gradually move from native nodes to Faust/WASM/Rust/native without rewriting the UI system again.
